Gear Library — quelle approche pour éviter les doublons ?

Deux points d'entrée : ajout manuel + import CSV. Trois façons d'attaquer le problème.

A

Smart add — suggestion avant création

Quand l'utilisateur tape un nom d'item (à l'ajout manuel), une dropdown apparaît avec les items existants de la library qui matchent. S'il choisit un existant → on le réutilise (référence). S'il continue → nouvel item. À l'import CSV, même logique : avant d'importer, un écran de "review" montre les doublons détectés et propose de les fusionner.

Pour

  • UX naturelle et non-intrusive
  • L'utilisateur garde le contrôle
  • Rétrocompatible complet

Contre

  • Matching par nom = fragile (casse, fautes)
  • Deux points d'entrée à implémenter
B

Library first — glisser depuis la library

On inverse le flux : au lieu d'ajouter un item dans une liste, on place un item de la library dans une liste (drag ou bouton "Add to list"). Créer un item directement dans une liste reste possible, mais la library devient le point de départ naturel. L'import CSV alimente la library, puis l'utilisateur choisit ce qu'il veut inclure.

Pour

  • Library = source de vérité
  • Zéro doublon par design
  • Cohérent avec PackStack/Lighterpack v1

Contre

  • Changement de paradigme UX
  • Flux d'import à repenser
  • Plus complexe à implémenter
C

Hybrid — suggestions + dédup automatique à l'import

À l'ajout manuel : autocomplete sur les items existants (comme A). À l'import CSV : déduplication automatique par fuzzy match sur le nom + poids — les doublons détectés sont mergés silencieusement, avec un résumé "3 items fusionnés" affiché après. L'utilisateur ne voit les conflits que s'il y a ambiguïté.

Pour

  • Meilleur des deux mondes
  • Import fluide sans friction
  • Ajout manuel guidé

Contre

  • Fuzzy match peut faire des erreurs
  • Scope plus large à implémenter
